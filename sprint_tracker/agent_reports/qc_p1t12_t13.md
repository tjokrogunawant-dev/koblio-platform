## P1-T12: Parent-Child Linking & School Association

**Verdict:** PASS-WITH-WARNINGS

### Findings

| Severity | Finding | File | Detail |
|---|---|---|---|
| WARNING | TOCTOU race in duplicate-enrollment check | `apps/api/src/user/user.service.ts` lines 271–291 | `enrollment.findUnique` (duplicate check) runs outside the transaction. Under concurrent requests two parents could enroll the same child simultaneously, bypassing the `ConflictException` and hitting the DB unique constraint, which would return an unhandled Prisma P2002 error (500). The DB schema has `@@unique([studentId, classroomId])` so data integrity is preserved, but the HTTP response would be 500 instead of 409. Fix: move the `enrollment.findUnique` check inside the `$transaction` callback, or add a global Prisma exception filter to map P2002 → 409. |
| WARNING | `getSchool` has no ownership check — any teacher/admin can query any school by UUID | `apps/api/src/user/user.service.ts` lines 312–336 | Only public aggregate data is exposed (name, country, counts) so no PII leak. Acceptable for MVP but noted for future when richer school data may be included. |
| WARNING | Missing `@ApiBearerAuth()` decorator on all new endpoints | `apps/api/src/user/user.controller.ts` lines 47–83 | Swagger docs will not show the Authorization header requirement. Pre-existing endpoints have the same gap (acknowledged in dev report). Carry-forward fix for whole controller. |
| INFO | `$transaction` wraps a single `enrollment.create` | `apps/api/src/user/user.service.ts` lines 284–291 | Functionally correct and consistent with codebase pattern. The transaction adds no isolation benefit when it wraps a single statement. Could be simplified, but this is purely cosmetic. |

### COPPA compliance

PASS. No `email` field appears in any child/student response shape in `getChild`, `joinClassByCode`, `listChildren`, or `createChildAccount`. The child `User` record is created without an email field (`tx.user.create` data object has no `email` key). The spec test at `user.service.spec.ts` line 224 explicitly asserts `args.data.email === undefined`. All child response objects return: `id`, `role`, `name`, `username`, `grade`, `country`, `classroom_id`, `linked_parent_ids` — no email.

### Security (IDOR, authorization)

PASS. Both `getChild` and `joinClassByCode` resolve the parent from the authenticated `auth0Id` from the JWT (`CurrentUser` decorator), then verify the `ParentChildLink` using `parentId_childId` composite key before returning or modifying child data. A parent cannot access another parent's child — the link lookup will return null and throw `NotFoundException`. The `ParseUUIDPipe` on `:childId` and `:schoolId` prevents invalid UUID injection. `RolesGuard` is a proper NestJS guard applied globally and enforces `@Roles()` metadata — parent-only routes are not accessible to teachers or students.

## P1-T13: Auth Frontend

**Verdict:** PASS-WITH-WARNINGS

### Findings

| Severity | Finding | File | Detail |
|---|---|---|---|
| WARNING | JWT access token stored in `localStorage` — XSS risk | `apps/web/src/lib/auth.ts` lines 7–13 | `localStorage.setItem(TOKEN_KEY, token)` stores the bearer token where any injected script can read it. For MVP this is a known and accepted trade-off (noted in dev report), but flagged per QC policy. Mitigation path: move to `HttpOnly` cookie storage before public launch. |
| WARNING | `koblio_session` cookie has no `Secure` flag | `apps/web/src/components/providers/auth-provider.tsx` line 65 | `document.cookie = 'koblio_session=1; path=/; SameSite=Lax'` — missing `; Secure`. On HTTPS deployments (Railway/Render) this is harmless in practice (browsers send the cookie over HTTPS regardless), but best practice is to include `Secure` explicitly. Low urgency for MVP. |
| WARNING | Only 2 tests exist; no tests for error states, redirects, or registration forms | `apps/web/src/__tests__/login-form.test.tsx` | The two tests cover tab-switching UI only (correct and well-written). No test covers: failed login showing inline error, loading state disabling the button, redirect after successful login, or any registration form. Minimum bar is met per task spec ("at least 2 tests"), but coverage is thin. |
| WARNING | `expires_in` fallback absent — token treated as expired if backend omits field | `apps/web/src/lib/auth.ts` lines 9–13, `apps/web/src/lib/api.ts` line 13 | `setStoredToken(token, expiresIn)` computes `Date.now() + expiresIn * 1000`. If `expires_in` is `undefined`, expiry is set to `Date.now() + NaN` which `isTokenExpired()` evaluates as `NaN > number` → `false`, meaning the token never expires client-side. This is a silent failure mode rather than an immediate break, but should be hardened with a fallback (e.g. `expiresIn ?? 3600`). Acknowledged in dev report. |
| INFO | `AuthUser` stored in `localStorage` alongside token | `apps/web/src/components/providers/auth-provider.tsx` lines 33–43 | User object (id, role, name, grade) is persisted to localStorage. No PII more sensitive than what's already in the JWT. Acceptable. |

### COPPA compliance

PASS. The student login path is entirely email-free. The `LoginForm` conditionally renders `<Input type="email">` only when `tab === 'adult'`. When the Student tab is active, only a `username` field is shown. The student dashboard (`apps/web/src/app/dashboard/student/page.tsx`) displays `username`, grade, and placeholder gamification stats — no email field anywhere. No student registration form exists (students are created by parents via the backend API, which is the correct COPPA pattern).

### Security

Token storage uses `localStorage` (WARNING above). No hardcoded API keys or secrets found in any source file. API base URL is correctly sourced from `process.env.NEXT_PUBLIC_API_URL` with a localhost fallback (`apps/web/src/lib/api.ts` line 1). Registration endpoints call `/auth/register/parent` and `/auth/register/teacher` respectively — matching the NestJS controller routes. No `alert()` calls; errors are displayed inline via `<p role="alert">` elements. Submit buttons use `disabled={loading}` to prevent double-submission. Middleware correctly guards `/dashboard/:path*` via `koblio_session` cookie check — unauthenticated users are redirected to `/login?next=<path>`.

Neither PR has any BLOCKER issues. All COPPA hard requirements pass. IDOR protection is solid on P1-T12. Auth flows are correctly gated on P1-T13.

### Required fixes before merge (BLOCKERs):

### Recommended fixes (WARNINGs):

1. **P1-T12 — Race condition in `joinClassByCode`** (`user.service.ts` lines 271–291): Move the duplicate-enrollment `findUnique` inside the `$transaction` callback, **or** add a global Prisma exception filter that maps `PrismaClientKnownRequestError` code `P2002` to `ConflictException` (409). Without this, concurrent enrollments can produce a 500.

2. **P1-T12 — Add `@ApiBearerAuth()` to `UserController`** (`user.controller.ts`): Apply to all authenticated endpoints so Swagger docs reflect auth requirements.

3. **P1-T13 — `expires_in` fallback** (`lib/auth.ts` line 11): Change `expiresIn * 1000` to `(expiresIn ?? 3600) * 1000` to handle backends that omit the field.

4. **P1-T13 — Add `; Secure` to `koblio_session` cookie** (`auth-provider.tsx` line 65): Harden before any production deployment.

5. **P1-T13 — Expand test coverage** for login error state and at least one registration form submission path.
