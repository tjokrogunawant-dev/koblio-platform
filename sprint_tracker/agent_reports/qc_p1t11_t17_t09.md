## P1-T11: Student Login + RBAC

**Verdict:** PASS-WITH-WARNINGS

### Findings
| Severity | Finding | File | Line |
|---|---|---|---|
| WARNING | `AuthResult` interface declares `user.email?: string` as an optional field. The `loginStudent()` return object correctly omits it, and the test asserts `'email' in result.user` is false. However, a future developer could accidentally add `email` to this return site without a type-level guard. Consider a separate `StudentAuthResult` type that does not include `email` at all. | `apps/api/src/auth/auth.service.ts` | 24–32 |
| WARNING | `JwtStrategy.validate()` maps `payload.email` into `AuthenticatedUser.email`. For internal student tokens, `email` will be `undefined` in the payload, so nothing is leaked at runtime. But if Auth0 ever includes an `email` claim in a student-impersonating token the field would propagate to `req.user`. Not a current risk; worth documenting explicitly. | `apps/api/src/auth/strategies/jwt.strategy.ts` | 78–80 |
| WARNING | `audience` and `issuer` claims are not validated at the Passport layer for Auth0 tokens (intentionally deferred). This is noted in the dev report and is acceptable for MVP, but should be tracked as a hardening item before Trial Gate 1. | `apps/api/src/auth/strategies/jwt.strategy.ts` | 57–60 |
| INFO | `StudentLoginDto.password` only enforces `@MinLength(6)`. No `@MaxLength` is set. Extremely long passwords (e.g., 10 KB) could slow bcrypt. A `@MaxLength(128)` decorator is recommended. | `apps/api/src/auth/dto/student-login.dto.ts` | 10–12 |
| INFO | `username` in `StudentLoginDto` has no maximum length or character-set validation. Consider `@MaxLength(20)` and `@Matches(/^[a-z0-9]+$/)` consistent with `generateUsername()` output. | `apps/api/src/auth/dto/student-login.dto.ts` | 6–8 |
| INFO | `generateUsername()` uses `Math.random()` (4-digit suffix). With many students sharing a base name, collision probability grows. A `ConflictException` is thrown on collision but there is no retry loop — the caller must try again. This is fine for MVP; add a retry loop before scale. | `apps/api/src/user/user.service.ts` | 191–197 |
| INFO | API `.env.example` in Dev 1 branch does not include `REDIS_URL`. The `RedisService` is already in use; new contributors need this variable documented. (Pre-existing gap, not introduced by this PR — flagged for awareness.) | `apps/api/.env.example` | — |

### COPPA compliance
**PASS.** The `loginStudent()` method returns `{ id, role, name }` — no `email` field. The test at line 431 of `auth.service.spec.ts` explicitly asserts `'email' in result.user` is `false`. The student `User` row has `email: null` by design (`createChildAccount` stores no email). There is no path in `loginStudent()` where an email value could be included in the response.

### Security
**PASS.** All critical security requirements are met:

- Password hashing: bcrypt with 10 salt rounds (`bcrypt.hash(dto.password, 10)` in `user.service.ts` line 63; `bcrypt.compare` in `auth.service.ts` line 283). No MD5/SHA/plain text.
- JWT signing: `JwtModule.registerAsync` uses `JWT_SECRET` from `ConfigService.getOrThrow` — will throw at startup if unset. `jwtService.sign()` uses HS256 by default with the configured secret. Algorithm `'none'` is not possible with `@nestjs/jwt` defaults.
- JWT payload includes `iss: 'koblio-internal'` (auth.service.ts line 292). The dual-strategy `secretOrKeyProvider` in `jwt.strategy.ts` routes tokens with this issuer to the symmetric `JWT_SECRET` and all others to Auth0 JWKS — preventing an internal token from being validated as an Auth0 token or vice versa.
- `ignoreExpiration: false` is set; expired tokens are rejected.
- `JWT_SECRET` is documented in `.env.example` with a guidance comment ("must be at least 32 chars").

## P1-T17: KaTeX Math Rendering

**Verdict:** PASS-WITH-WARNINGS

### Findings
| Severity | Finding | File | Line |
|---|---|---|---|
| WARNING | The `MathRenderer` component carries `'use client'` and imports `katex/dist/katex.min.css`. This CSS import will work fine in a Next.js app, but consuming this component from `@koblio/ui` in a plain Node/SSR test environment (without CSS mocking) will fail. The test file correctly mocks it via `jest.mock('katex/dist/katex.min.css', () => ({}))`, and the dev report acknowledges this. However, there is no jest config in `packages/ui` itself to handle this mock — consumers of the package running their own tests would need to add the mock. Consider moving the CSS import to the consuming app, or documenting the required jest transform. | `packages/ui/src/math-renderer.tsx` | 5 |
| WARNING | `react-katex@^3.0.1` is unmaintained (last release 2021, open issues unresolved). This is acknowledged in the dev report. For MVP it is acceptable, but it should be tracked for replacement with direct `katex` usage before the Flutter/web content delivery phase. | `packages/ui/package.json` | 49 |
| WARNING | `katex` and `react-katex` are added to both `packages/ui/package.json` (dependencies) and `apps/web/package.json` (dependencies). In a pnpm monorepo this is fine but creates a potential for version skew. Consider keeping `katex`/`react-katex` only in `packages/ui` and letting the app consume it through the package. | `apps/web/package.json` | 24–29 |
| INFO | The error boundary uses `getDerivedStateFromError` but does not implement `componentDidCatch`. This means rendering errors are silently swallowed with no logging. Adding a `componentDidCatch` that calls `console.error` (or forwards to Sentry once P1-T09 is live) would aid debugging. | `packages/ui/src/math-renderer.tsx` | 25–27 |
| INFO | Tests live in `apps/web/src/__tests__/` but the component lives in `packages/ui`. This is reasonable for an integration-style test but there are no unit tests in `packages/ui` itself. Low-priority since the web tests cover the same component. | `apps/web/src/__tests__/math-renderer.test.tsx` | — |
| INFO | The test does not cover the error boundary fallback path (simulating a render error from `react-katex` and confirming the `<code>` fallback renders). This is the highest-value untested code path. | `apps/web/src/__tests__/math-renderer.test.tsx` | — |

`'katex/dist/katex.min.css'` is imported directly in `math-renderer.tsx` line 5. This will render KaTeX with correct styling in the web app. The test mocks it correctly.

### TypeScript types

## P1-T09: Sentry Error Tracking

**Verdict:** PASS

### Findings
| Severity | Finding | File | Line |
|---|---|---|---|
| INFO | `profilesSampleRate: 1.0` on the API instruments every request in all environments including production. At MVP scale this is fine, but once traffic grows this will add overhead. The dev report acknowledges this is intentional for baseline collection; it should be made environment-aware before Trial Gate 1 (e.g., `process.env.NODE_ENV === 'production' ? 0.1 : 1.0`). | `apps/api/src/instrument.ts` | 9 |
| INFO | `withSentryConfig` is called with `silent: true`, which suppresses all Sentry CLI build output. This is reasonable for CI noise reduction but makes it harder to debug source map upload failures. Consider setting `silent: process.env.CI === 'true'` so local builds show output. | `apps/web/next.config.ts` | 9 |
| INFO | `sentry.client.config.ts` sets `debug: false` explicitly. The other two config files (`sentry.server.config.ts`, `sentry.edge.config.ts`) do not set `debug`. This is a minor inconsistency — all three should agree. | `apps/web/sentry.client.config.ts` | 7 |

### DSN / secrets check
**PASS.** All three web Sentry configs read DSN from `process.env.NEXT_PUBLIC_SENTRY_DSN`. The API `instrument.ts` reads `process.env.SENTRY_DSN`. No hardcoded DSN strings or org tokens found anywhere. Both `.env.example` files have the correct placeholder entries (`NEXT_PUBLIC_SENTRY_DSN=` and `SENTRY_DSN=`).

### Import order check
**PASS.** `import './instrument'` is the first line in `apps/api/src/main.ts` (line 1), before any NestJS imports. This is the required order for Sentry to instrument the Node.js runtime correctly.

### Module setup check
**PASS.** `SentryModule.forRoot()` is the first entry in the `AppModule.imports` array (`app.module.ts` line 21).

### `withSentryConfig` check
**PASS.** `next.config.ts` wraps `nextConfig` with `withSentryConfig`. `org` and `project` are read from environment variables; if unset, the Sentry CLI skips source map upload silently (safe default for local dev).
