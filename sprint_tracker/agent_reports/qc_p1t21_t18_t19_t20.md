## P1-T21: Attempt Recording Backend

**Verdict:** PASS-WITH-WARNINGS

### Findings

| Severity | Finding | File | Detail |
|---|---|---|---|
| WARNING | No upper-bound guard on pagination `limit` query param | `attempt.controller.ts` line 40–41 | A caller can pass `limit=999999` to `GET /attempts/me`, triggering a huge DB scan. The service defaults to 20 but accepts whatever the controller passes without capping it. Add `Math.min(limit ?? 20, 100)` in the controller or service. |
| WARNING | `getStudentStats` fires 3 parallel queries including `findMany` with `distinct` — no pagination | `attempt.service.ts` lines 104–115 | For students with thousands of attempts the `findMany` (used only to extract topics) could be very large. At MVP scale this is acceptable, but worth noting for Section 8+ adaptive engine work. |
| INFO | `correctAnswer` is returned in the submit response | `attempt.service.ts` line 58 | The correct answer is sent back on every submission, even before the student has answered (the client receives it in the response body). This is intentional per the design (show solution after submit), confirmed by the frontend using it in the ANSWERED state — not a bug. |

### Security: cross-student isolation

PASS. Every service method accepts `studentId` as a parameter derived exclusively from `user.userId` (the JWT claim) in the controller — never from request body or query string. All `findMany`/`count` calls scope on `{ studentId }`. No endpoint allows querying another student's data.

### Role enforcement

PASS. All 4 endpoints carry `@Roles('student')` (lines 24, 34, 57, 64 of `attempt.controller.ts`). No teacher or parent can call these endpoints.

### Answer validation

PASS. `submitAnswer` fetches the problem from the DB (lines 25–33) and derives `correct` server-side (line 40–41). The client cannot influence the `correct` flag. Case-insensitive comparison via `.trim().toLowerCase()` on both sides — correct.

### `timeSpentMs` bounds

PASS. DTO applies `@Min(0) @Max(600000)` (10 minutes) — lines 8–9 of `submit-answer.dto.ts`. Values outside this range will be rejected by class-validator before reaching the service.

### Migration SQL

PASS. Both required indexes are present:
- `student_problem_attempts_student_id_problem_id_idx` (line 14)
- `student_problem_attempts_student_id_created_at_idx` (line 15)

Foreign key constraints with `ON DELETE CASCADE` are present for both `student_id → users` and `problem_id → problems`.

### Schema back-relations

PASS.
- `User` model has `attempts StudentProblemAttempt[]` relation (schema.prisma line 44).
- `Problem` model has `attempts StudentProblemAttempt[]` relation (schema.prisma line 163).

### Test count: 9 (not 13 as claimed)

WARN. The spec file contains **9 `it(...)` blocks**, not 13:

| Block | Test |
|---|---|
| submitAnswer | correct=true when answer matches |
| submitAnswer | correct=false, attempt still created |
| submitAnswer | case-insensitive comparison |
| submitAnswer | NotFoundException when problem not found |

The implementation report claimed 13 tests. The actual count is 9. Coverage of the 4 core service methods is solid. Missing test cases: (1) pagination capping behaviour, (2) correct answer is never sourced from the DTO, (3) stats when `totalAttempts > 0` but `correctAttempts = 0` (0% accuracy branch with data present), (4) `getStudentAttempts` default limit/offset values. These are gaps but not blockers at this stage.

## P1-T18/T19/T20: Student Problem-Solving Frontend

**Verdict:** PASS-WITH-WARNINGS

### Findings

| Severity | Finding | File | Detail |
|---|---|---|---|
| WARNING | Correct answer shown only when `!isCorrect` | `learn/problem/[id]/page.tsx` lines 402–415 | The "Correct answer:" line is inside `{!isCorrect && result && ...}`. When the student answers correctly, the correct answer is not displayed. This is reasonable UX for correct answers, but the solution card (which follows) still appears — so the student sees the explanation. Minor UX inconsistency: a student who guesses correctly on a multi-step MCQ never sees which option was canonical. Not a blocker. |
| WARNING | TRUE_FALSE submits "True"/"False" (title-case) but backend stores/compares lowercase | `learn/problem/[id]/page.tsx` line 191 | The frontend calls `handleTrueFalse('True')` or `handleTrueFalse('False')`. The backend compares case-insensitively (`toLowerCase()`), so this is functionally correct. However, if a problem's stored answer is `"true"` and the student submits `"True"`, the comparison works correctly — confirmed. No bug. Documenting for awareness. |
| WARNING | Token passed as empty string `''` when unauthenticated | `learn/problem/[id]/page.tsx` line 157 | `submitAnswer(..., token ?? '')` — if `token` is null (unauthenticated user), the function sends `Authorization: Bearer ` with an empty token rather than omitting the header. The backend will reject with 401, which triggers the graceful-degradation fallback. Functionally safe, but cleaner to short-circuit before calling the API or to omit the header when token is absent. |
| INFO | `handleTryAgain` resets to `QUESTION` state without re-fetching the problem | `learn/problem/[id]/page.tsx` lines 204–212 | The timer restarts correctly (via the `pageState === 'QUESTION'` effect), `hintIndex` resets, and all input fields clear. Correct behaviour. |

### COPPA compliance

PASS. No email fields are rendered anywhere in the learn flow (`/learn/page.tsx` or `/learn/problem/[id]/page.tsx`). The only user-visible identifier is the auth token used invisibly for API calls. No PII is collected or displayed in these pages.

### Topic browser navigation (3-step flow)

PASS. Three distinct steps are implemented: `'grade'` → `'topics'` → `'problems'` (step state machine, `learn/page.tsx` lines 11, 69). Breadcrumb nav reflects current step. Back buttons on both inner steps are present and functional.

### Problem type coverage

PASS. All 3 types handled in `learn/problem/[id]/page.tsx`:
- MCQ: option buttons rendered from `problem.options` (lines 309–329)
- FILL_BLANK: text input + form submit (lines 331–349)
- TRUE_FALSE: True/False button pair (lines 352–365)

### Timer

PASS. Timer starts when `pageState === 'QUESTION'` (lines 113–124), uses `setInterval` with 1s resolution for display, and `stopTimer()` captures the precise `Date.now() - startTimeRef.current` value at submit time (lines 133–139, 146). The precise elapsed value (not the rounded display value) is sent to the API.

### Hint system

PASS. Progressive hints via `hintIndex` state (starts at -1). Each press of the Hint button increments `hintIndex` by 1 and shows all hints up to that index (lines 195–202, 292–305). Button disables when all hints exhausted (line 260). `hintUsed` flag is set on first hint reveal (line 200) and passed to `submitAnswer` (line 155).

### Answer submission

PASS. `handleSubmit` calls `submitAnswer` from `api.ts` with `{ problemId, answer, timeSpentMs, hintUsed }` (lines 154–156). The `submitAnswer` function in `api.ts` posts to `POST /attempts` with Bearer token (lines 184–197).

### Graceful degradation

PASS. On API failure, the `catch` block (lines 164–171) sets `submitError` for a non-blocking warning banner, falls back to `localCorrect` (computed server-side-equivalent comparison on line 151), and uses `problem.correctAnswer` / `problem.solution` from the cached problem object. `setPageState('ANSWERED')` runs in `finally` so the student always sees a result.

### Correct/incorrect feedback

PASS. Green banner with "Correct!" or red banner with "Not quite." displayed prominently (lines 389–416). Incorrect answers also show the student's submitted answer and the correct answer.

### Solution text

PASS. Solution card rendered when `result?.solution` is truthy (lines 426–435), using `RichText` which supports inline LaTeX via `MathRenderer`.

### "Try Again" button

PASS. Present for incorrect answers (line 457–462). Resets `pageState` to `'QUESTION'`, clears all input/hint/result state (lines 204–212). Timer restarts via the `useEffect` on `pageState`.

### `submitAnswer` in `api.ts`

PASS. Function exists at lines 184–197 of `apps/web/src/lib/api.ts` with correct types (`SubmitAnswerData`, `SubmitAnswerResponse`).

### Test count: 4 (2 files, 2 tests each)

PASS (minimum threshold met). 2 tests in `learn-topic-browser.test.tsx`, 2 tests in `learn-problem-page.test.tsx` = 4 total.

Coverage gaps (not blockers): no tests for FILL_BLANK or TRUE_FALSE rendering, no test for API failure + graceful degradation, no test for hint button progression, no test for Try Again reset. These should be added in a follow-up sprint.

3. **`attempt.service.spec.ts`** — Reported test count (13) does not match actual count (9). Update the implementation report or add the 4 missing test cases before Sprint 05 retro.

4. **Frontend tests** — Add at least one test each for: FILL_BLANK rendering, TRUE_FALSE rendering, API failure + graceful degradation path, hint progression.
